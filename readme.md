## 第一阶段

### 基本需求
实现五子棋和围棋的对战功能, 可以进行玩家-玩家的双人对战, 双方交替落子, 五子棋和围棋规则可自行搜索. 支持基本的局面管理和终局判断功能, 可以处理用户的不合法操作, 支持局面的保存与读取功能. 要求体现面向对象设计模式, 保证类的可复用性和可扩展性, 降低类之间的耦合程度. 

### 局面管理与终局判断
- 两名玩家分别执黑白棋, 交替落子. 
- 围棋行棋过程中需要支持提子, 即提出棋盘上无气的棋子, 且下子时不能下到不合法位置. 围棋行棋过程中可以选择不落子(虚着). 
- 围棋在下到无子可下或双方均决定不落子时由系统判断胜负, 判断胜负时采用任何一种现行的胜负判断方法均可. 
- 五子棋中若一方连成五子, 则系统自动判断获胜者并结束游戏, 若棋盘被占满则输出平局. 
- 一方投子认负后棋局立即结束.

### 界面与交互
- 使用控制台命令行实现即可, 键盘输入功能指令. 
- 需要支持以下功能指令：
    1. 在游戏未开始或结束时开始游戏, 选择游戏类型, 同时设定棋盘的大小(最小8\*8, 最大19\*19)
    2. 在游戏对局过程中重新开始游戏
    3. 在棋盘指定位置落子或不落子(不落子仅限围棋)
    4. 悔棋一步
    5. 投子认负
    6. 将当前局面保存在指定文件
    7. 从指定文件读取存档覆盖当前局面
- 控制台中需显示对局必要的信息, 能明显区分行棋双方的棋子, 对于用户的所有指令都需要给出反馈. 
- 界面上需显示给用户的操作提示, 用户可以选择隐藏和重新显示提示

### 设计要求
- 平台后端(负责游戏状态维护、逻辑判断)与客户端(负责接受用户指令、将指令传递到平台后端执行、执行结果反馈给用户)功能分离, 客户端可以以同样的形式对待不同的游戏类型
- 尽可能减少对象创建和销毁的开销, 尽可能不使用全局变量
- 交互界面需用先设计后建造的方式来显示, 界面中包含不同的组件
- 系统需要能处理用户的异常输入并给出错误提示, 需自行决定应在后端还是客户端判断异常
  1. 无棋子可悔或悔棋次数超过限制
  2. 落子位置为不合法位置
  3. 保存局面失败或读取局面不合法
  4. 开始游戏时设置棋盘大小不符合要求
  5. 用户输入了不合法的指令
  6. 其他异常情况需自行考虑

### 可选功能
实现图形GUI界面:
- 棋盘和棋子有图片画面显示
- 支持鼠标点击完成各项交互操作
- 要求显示逻辑和业务逻辑分离


## 第二阶段

### 基本需求

在五子棋和围棋对战的基础上实现黑白棋的对战功能, 增加可以自动落子的AI功能, 增加支持玩家-AI或AI-AI的对战功能, 增加录像与回放功能, 增加用户账户管理功能.

要求体现面向对象设计模式, 保证类的可复用性和可扩展性, 降低类之间的耦合程度.

### 黑白棋

* 黑白棋初始大小推荐为 8 $\times$ 8.

* 行棋过程中需要注意只允许落子在合法位置而且会将对手被夹住的棋子翻转, 且注意当一方没有合法棋步可下时这回合将被迫弃权.

* 棋盘填满或者双方都无合法棋步可下时系统判断胜负.

* 其他交互需求与基本需求一致.

### 录像与回放

* 在行棋过程中可以记录用户每一步行棋的过程, 保存局面时同时可以记录局面的变化过程.

* 在读取存档的基础上增加回放功能, 即增加一个回放模式可以在局面上观看存档中双方每一步的行棋过程.

### AI功能

* 增加可以自动落子的 AI, 在 `玩家-玩家` 对战的基础上可以选择进行 `玩家-AI` 和 `AI-AI` 对战的功能.

* 对于五子棋或黑白棋的一种, 实现至少两个级别的AI算法：

- 在合法位置随机落子的简单AI (一级AI).

- 简单的人工规则或评分函数, 要求可以稳定战胜一级AI (二级AI).

- **可选功能 (+10%)**: 使用蒙特卡洛树搜索 (MCTS) 方法或强化学习方法实现更强的AI (三级AI), 要求至少可以稳定战胜二级AI, 且在与玩家对战过程中呈现足够的智能.

### 用户账户管理

* 两位玩家可以在对弈前输入用户名和密码登录自己的账号, 若没有账号则可以注册账号, 平台需要记录每个账号的战绩 (对战场次, 胜场), 账号信息保存于本地文件.

* 对弈过程中界面需要额外展示玩家的用户名和战绩, 未登录的玩家显示为游客, AI 角色显示为 AI.

* 在一次对局结束后, 平台系统需要更新登录玩家的战绩信息.

* 用户保存的录像文件与用户的账号信息关联.

### 可选功能: 网络对战 (+10%)

* 支持两个客户端在局域网内通过网络进行对战, 双方可选择先后手 (也可在一台电脑上开两个客户端通过本地网络通信).

* 棋盘的局面和指令需要在两个客户端间同步.

### 设计要求

* 按照面向对象各项设计原则进行设计, 需明确使用的设计模式.

* 增加黑白棋游戏类型后, 类接口和设计尽可能少发生变化, 不能破坏第一阶段已经实现好的功能.

* 平台系统可以无差异地对待对弈双方是玩家或是AI的情况, 包括未登录的玩家、登录的玩家、不同级别的AI算法.

* 代码逻辑尽可能清晰, 关键代码需有必要的注释


代码要求符合面向对象编程原则, 体现课上所学的设计模式, 具备良好的可读性和可扩展性.

设计文档需要包括设计思路和设计模式选用、关键类和函数功能说明(以及实现过程中如何体现面向对象的设计原则)、UML图、测试输入和输出结果.
